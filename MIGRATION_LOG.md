# Migration Log: Claude → Local Gemma 4 E2B

**Date:** 2026-07-08
**Executed per:** `Plan-of-Migration.md`
**Result:** Complete. Default LLM backend is now local Gemma 4 E2B (llama-server);
Claude remains available via `LLM_BACKEND=claude`. Fully reversible.

---

## Key finding (differs from the plan's assumption)

The plan assumed LangChain `ChatAnthropic` was the most common call form. It is
**not present**. MetaMAVS uses the **official Anthropic SDK**, and it was already
**fully centralized** in one module — `metamavs/llm/client.py` — behind four
functions (`generate`, `generate_json`, `llm_available`, `resolve_params`) that
all six LLM agents call. This is the "Situation B (official SDK)" path.

Consequence: the cleanest migration adds an OpenAI-compatible backend **inside
`client.py`**, behind an `LLM_BACKEND` switch, rather than introducing a separate
LangChain-based `llm_factory.py`. This keeps the existing interface and required
**zero changes to any agent's business logic**.

## Environment

- Endpoint: `http://localhost:8080/v1` (llama-server, `--jinja`)

- API key: not needed (`GEMMA_API_KEY` defaults to `"not-needed"`)

## Problems found & solutions

1. **Gemma is a reasoning model.** With small `max_tokens`, hidden reasoning
   consumed the entire budget and `message.content` came back empty
   (`finish_reason: length`). *Fix:* a `GEMMA_MIN_MAX_TOKENS = 2048` floor in the
   gemma path, plus a warning when content is still empty. Narrative agents use
   `max_tokens=4000`, which leaves ample room after reasoning.
2. **No cache-breakpoint concept in the OpenAI chat API.** The shared reference
   (a Claude `cache_control` breakpoint) is folded into the system message;
   llama-server still prefix-caches it automatically (observed `cached_tokens`).
3. **Wrong model attribution in reports.** `resolve_params` returns the configured
   (Claude) model string, which is a no-op under gemma, so reports read
   "Generated by claude-opus-4-8". *Fix:* `active_model()` returns the real served
   model; the interpretation agent stores that. Reports now read
   "Generated by gemma-4-E2B-it".

## Quality assessment (Task 3.4)

Gemma's surveillance narrative was well-structured and scientifically cautious,
matching the project's caution rules: "detected sequence signal" (not "confirmed
infection"), phages reported separately, orthogonal confirmation recommended for
all high-risk detections. Per-taxon risk reasoning was coherent. Latency ~5–8 s per
narrative call on this host. Judged **acceptable** for the surveillance workflow;
Claude remains available for cases needing stronger reasoning.

## Rollback

- Quick: `git revert` the migration commit, or
- Gradual: `export LLM_BACKEND=claude` (+ `ANTHROPIC_API_KEY` in `.env`) — no code
  change, immediate.

## Follow-up suggestions

- Add a config-level `llm.backend` mirror of `LLM_BACKEND` for single-file control.
- Consider a lightweight retry/JSON-repair loop for the `generate_json` agents if
  Gemma occasionally emits malformed JSON under load.
- Optionally monitor/auto-restart llama-server (systemd/supervisor) for unattended runs.
